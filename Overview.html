Intro

En el ecosistema de desarrollo moderno, es crucial entender el contexto y propósito de cada servicio API. Más allá del consumo técnico, estas APIs representan funcionalidades críticas para el negocio, y conocerlas a fondo facilita su mantenimiento, evolución y monitoreo.

Esta sección sirve como guía general para mapear, clasificar e identificar dependencias funcionales y técnicas entre servicios, ayudando tanto a desarrolladores como testers a comprender cómo interactúan las piezas de un sistema distribuido.

Tipo de Servicio

Clasificar las APIs según su propósito ayuda a definir mejores prácticas de consumo y pruebas. A continuación se detallan algunos tipos comunes:

  • Interno: De uso exclusivo dentro de la organización.
  • Público: Disponible para consumo por terceros o desarrolladores externos.
  • De Apoyo: Soportan procesos auxiliares como logs o validaciones.
  • De Reporte: Devuelven datos consolidados o históricos.
  • De Creación de Datos: Permiten insertar nuevos registros en el sistema.
  • De Revisión de Datos: Permiten consultar registros existentes.
  • De Autenticación: Manejan el login y control de acceso.
  • De Guardado de Datos: Envían información persistente a la base de datos.

Mapa de Servicios

El mapa de servicios permite visualizar cómo se interconectan las APIs entre sí y con otras aplicaciones. Esto facilita la comprensión del flujo de datos, puntos críticos y la identificación de posibles cuellos de botella.

Idealmente, este mapa se presenta como un diagrama o esquema visual, pero también puede describirse como una jerarquía funcional: por ejemplo, un microservicio de usuarios depende de servicios de autenticación, perfiles y notificaciones.

Dependencias

Las aplicaciones modernas rara vez funcionan de forma aislada. Una API puede depender de otras para validar datos, obtener configuraciones o procesar respuestas en cadena. Es fundamental documentar estas dependencias explícitamente.

Esto permite prever cómo una caída en un servicio puede afectar a otros, o cómo una actualización en la API origen puede requerir ajustes en la API dependiente. En entornos de pruebas, también permite simular mejor los escenarios reales.

Mecanismo de Cache

El uso de cache permite mejorar el rendimiento de las APIs reduciendo la latencia en consultas repetidas. Dependiendo del caso de uso, se pueden aplicar mecanismos de cache temporales o permanentes.

Algunos ejemplos populares incluyen:

  • Redis: Almacena pares clave-valor con tiempo de expiración. Ideal para tokens, sesiones y respuestas frecuentes.
  • CDN (Content Delivery Network): Usado para cachear respuestas estáticas o datos de consulta pública en servicios como Cloudflare o Akamai.
  • Cache-Control: Header HTTP que permite controlar desde el cliente o servidor el tiempo de validez de una respuesta.

Observabilidad

La observabilidad permite comprender el comportamiento interno de una API con base en sus métricas, logs y trazas. Un buen sistema debe permitir responder preguntas como: ¿funciona correctamente la API?, ¿dónde está fallando?, ¿cuáles son los tiempos de respuesta?

Se recomienda utilizar herramientas como Grafana, Prometheus, ELK Stack (Elasticsearch, Logstash, Kibana) o New Relic para monitorear eventos clave. La trazabilidad entre microservicios también puede lograrse con OpenTelemetry y Jaeger.

Esto facilita la detección temprana de errores y el análisis post-mortem ante incidentes en producción.
